# Support

Clawsembly is pre-release infrastructure. The public compatibility report is
the source of truth for what is currently proven; `probing` does not authorize
a production BrowserPod launch.

## Where to ask

- Use [Q&A Discussions](https://github.com/haya-inc/clawsembly/discussions/categories/q-a)
  for how-to questions and help interpreting a report before opening an issue.
- Use the support request template for a reproducible integration or install
  blocker with exact versions and redacted evidence.
- Use [Ideas](https://github.com/haya-inc/clawsembly/discussions/categories/ideas)
  to test a proposal and its evidence boundary before implementation.
- Use the bug template for reproducible defects in Clawsembly itself.
- Use the compatibility template for exact OpenClaw or BrowserPod findings.
- Use [issue #9](https://github.com/haya-inc/clawsembly/issues/9) to share an
  external policy, report, badge, Action, or SDK integration.
- Use [Show and tell](https://github.com/haya-inc/clawsembly/discussions/18) to
  present that integration to other adopters and collect feedback.
- Report security issues privately according to [SECURITY.md](SECURITY.md).

Include the exact Clawsembly release or commit, OpenClaw version, BrowserPod
version, browser and operating system, and a minimal redacted reproduction.
Never post API keys, Gateway or device tokens, private keys, user payloads, or
workspace contents.

Community support has no response-time guarantee. Complete private security
reports follow the acknowledgement target in the security policy.
