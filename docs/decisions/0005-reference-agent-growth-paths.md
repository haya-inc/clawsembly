# ADR 0005: The reference agent demonstrates the two growth paths

- Status: accepted
- Date: 2026-07-13

## Context

The owner's 2026-07-13 direction: create the situation where something that
behaves like OpenClaw runs browser-locally now, and where the embedded agent
can be grown from the inside or extended from the outside — without waiting
on the upstream or the runtime vendor.

Three facts shaped the response:

- **OpenClaw runtime evidence is vendor-blocked.** BrowserPod 2.12.1
  provisions Node 22.15.0, below the 22.19 baseline pinned by
  `openclaw@2026.6.11`, so the readiness probe fails closed
  ([issue #6](https://github.com/haya-inc/clawsembly/issues/6)). The bound
  real upstream cannot boot on the committed runtime today, for a reason
  outside this repository's control.
- **The first reference binding proved portability, not extensibility.** The
  hello-agent binding landed with one greeting method and a deliberately
  empty capability-requirement declaration. No guest ever crossed the
  capability mailbox, so "the embedder extends an agent through the
  boundary" — the durable value chosen in ADR 0004 — remained a stated
  property rather than a demonstrated one.
- **The boundary's product story is exactly a growth story.** ADR 0004
  positions the embedder-controlled host boundary as the reusable product.
  What an embedding application buys from that boundary is optionality: grow
  the embedded agent, or extend what it may do, on the embedder's terms.

## Decision

The hello-agent reference binding grows from a greeter into a minimal
capability-consuming chat agent and becomes the demonstration vehicle for
both growth paths.

1. **OpenClaw-shaped bounded surface.** The fixture protocol
   (`clawsembly-hello/2`) adds `chat.send`, `chat.history`, and `chat.abort`
   beside `hello.say`, with a streamed delta/done event shape, bounded
   payloads, and an in-flight abort — the same movement the OpenClaw
   embedding surface exposes (send, stream, history, abort), kept
   deliberately small.
2. **The external extension path is the only work path.** The agent's chat
   turn is delegated to the host capability `chat.complete` (scope
   `provider:reference`) through the staged, digest-pinned mailbox client;
   the agent holds no provider access of its own. Handlers, grants,
   expiry, revocation, and payload-free audit stay host-side. Denial is a
   visible, actionable failure (`capability_denied`), not a silent stub —
   approving, revoking, or re-pointing the handler changes what the agent
   can do without touching agent code.
3. **The internal growth path is the change itself.** The fixture is
   first-party canonical source; growing it required a version bump
   (0.1.0 → 0.2.0), a regenerated byte-reproducible tarball identity, and
   updated per-file digests, and every stale pin fails
   `npm run hello-agent:check`. Growing the embedded agent is a normal,
   evidence-disciplined release motion, not a fork of the machinery.
4. **The evidence gate hardens with the surface.** Valid runtime evidence now
   requires a live capability transport at readiness, hello and chat round
   trips, and at least one denied and one allowed capability outcome — so a
   record cannot claim the boundary was exercised unless default-deny and an
   explicit grant were both observed.
5. **The reference agent stays bootable on today's vendor runtime.** The
   fixture pins Node `>=22.12.0` and uses no API beyond that floor, so the
   complete verified chain — digest-verified staging, dual readiness, a
   capability-mediated chat turn, cooperative stop — is capturable on the
   Node 22.15.0 that BrowserPod provisions now, independent of the OpenClaw
   baseline blocker.

Honesty constraints, unchanged from the binding contract: hello-agent is not
a real agent and not a product surface; it claims no verified runtime
support; it never appears in published reports or on Pages; OpenClaw remains
the only bound real upstream; provider-free test evidence is contract
evidence, not runtime support.

## Consequences

- The binding contract's capability-requirements clause now has a non-empty,
  executed demonstration; upstream portability and external extensibility are
  both test-demonstrated properties of the unmodified core.
- The two growth paths are demonstrable without metered tokens, and an
  owner-authorized BrowserPod capture of the reference binding becomes the
  first achievable real browser-local verified agent run while OpenClaw
  remains vendor-blocked. Such a capture proves the machinery; it does not
  and must not promote any OpenClaw report.
- Accepted risk: a chat-capable reference agent sits closer to "a second
  agent" than a greeter did. The strategy line is unchanged — Clawsembly does
  not become a second OpenClaw implementation — so reference-agent growth
  stays bounded to what demonstrating the boundary requires.

## Alternatives considered

1. **Grow the reference agent toward OpenClaw feature parity.** Rejected:
   `docs/oss-strategy.md` forbids becoming a second OpenClaw implementation,
   and parity is the upstream's job.
2. **Demonstrate the boundary with an in-page (non-guest) demo agent.**
   Rejected: it would bypass the runtime boundary, staging verification, and
   the mailbox transport — demonstrating a weaker claim than the one that
   matters.
3. **Wait for the vendor Node upgrade and demonstrate with OpenClaw.**
   Rejected: it leaves both growth paths undemonstrated for an unbounded
   vendor timeline and keeps the project's story hostage to issue #6.

## Related decisions

- ADR 0004 chose the upstream-portable boundary as the durable value; this
  ADR makes its extension path executable.
- ADR 0002's browser-local invariant and runtime acceptance gates are
  unchanged.
- The [upstream binding contract](../upstream-binding-contract.md) documents
  what the grown binding supplies.
