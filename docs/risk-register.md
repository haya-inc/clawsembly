# Risk register

| Risk | Current evidence | Decision or mitigation | Gate |
| --- | --- | --- | --- |
| Commercial browser-runtime terms | BrowserPod is the only runtime and explicitly permits commercial use on Pro/Enterprise plans, but is proprietary, metered, and vendor-hosted | Keep runtime terms visible, require explicit user opt-in/API key, obtain the appropriate plan before commercial beta, and keep the runtime contract narrow | Before public hosted beta |
| Hosted runtime dependency | BrowserPod's npm entry point loads a versioned runtime module from vendor infrastructure unless Enterprise self-hosting is contracted; the current project-page CSP does not authorize that origin | Keep the adapter dependency-injected, require the embedding host to review and explicitly authorize the pinned vendor delivery origin, document availability, privacy, caching, and outage behavior, and never claim fully self-contained operation for this provider | Phase 3b |
| Open runtime maturity | The pinned container2wasm v0.8.4 Node 22.19.0 probe took 1,625.84 s to produce a 316,700,841-byte module; Wasmtime v33.0.2 then exited 1 before running the guest command, including after a 512 MB debug rebuild reached the OCI root mount during snapshot creation | Keep the candidate experimental, resolve a minimal host guest-command gate before spending on `--to-js`, then publish Chromium transfer and cold-start measurements or reject it | Phase 3b |
| Runtime redistribution compliance | container2wasm output includes emulator, Linux, and guest packages with licenses beyond the converter's Apache-2.0 license | Generate an SBOM, notices, and corresponding-source workflow; complete legal review before distributing a generated image | Before distributing Wasm images |
| Vendor API surface | BrowserPod's public process/terminal types expose no input or termination; its documented filesystem API is sufficient for the typed polling mailbox | Keep terminal input unavailable, use bounded monotonic mailbox slots for capability calls and cancellation, and prove portal lifecycle and secret boundaries before support | Phase 3b |
| BrowserPod teardown gap | BrowserPod 2.12.1 documents `run()` completion, output terminals, and portals, but its published `Process` and `Terminal` types expose no input, termination, or Pod-disposal methods; the evidence Gateway now exits through a nonce-bound guest supervisor | Keep `interactiveInput`, `processTermination`, and `hardDispose` false; limit cooperative stop claims to Clawsembly-launched children; require a documented provider mechanism for arbitrary termination and Pod disposal | Phase 3b |
| Filesystem mailbox tampering | The untrusted guest can read and modify its persistent workspace, including mailbox manifests, staged client modules, payloads, and old slots | Use a fresh random channel per boot, trust only the in-memory broker, generate and digest-pin the client from canonical source, verify byte-exact staging, require exact subjects and monotonic slots, reject duplicate IDs/traversal/extra fields, bound both directions, and never transport host-only secrets | Phase 3c |
| Public portal exposure | BrowserPod portals are shareable public URLs rather than origin-local loopback endpoints | Treat portal URLs as public metadata, retain Gateway authentication, restrict allowed origins, never place secrets in portal URLs, and document revocation/session expiry | Before BrowserPod Gateway evidence |
| Runtime-wrapper commoditization | BrowserPod already demonstrates browser AI coding agents, so merely booting OpenClaw is not differentiated | Make exact compatibility evidence, the host capability broker, and evidence-bound embed manifests the public product surfaces; measure external report or SDK consumption | Every milestone |
| Node runtime mismatch | The BrowserPod probe contract requires Node 22.19+, but no owner-authorized provider record is checked in; one nested dependency warns that it prefers Node >=24 | Fail launch without matching preflight evidence, keep the warning visible, and exercise the code path before upgrading the browser baseline | Continuous |
| Artifact weight | After a 4.1% script-suppression improvement, Chromium measures 57.1 s cold install, 2.9 s warm reinstall, 618.5 MB `node_modules`, and 261.6 MB npm cache; the nested repair alone takes 49.7 s, while `npm ci` rejects 31 manifest dev dependencies missing from the shrinkwrap root | Keep the automated root-consistency warning, report the shrinkwrap defect upstream, cache or replace the 293-package repair path, and set regression budgets before beta | Phase 1 |
| Native dependencies | node-pty and sqlite-vec platform variants are present | Classify eager versus optional imports and disable only the owning capability | Phase 1 |
| Device identity extraction | A non-extractable browser Ed25519 key completes local Control UI pairing; the issued token is encrypted in IndexedDB and performs a token-only reconnect, while existing briefly in a dedicated bridge process | Keep the private key browser-only and the bridge ephemeral; add remote approval, rotation, revocation, recovery, and bridge hardening before user-facing auth | Before user-facing auth |
| Provider-secret exposure | An OpenClaw agent completes a turn through the browser-host broker without receiving the API key; the live smoke-test gate requires explicit consent, but same-origin script injection could still invoke the stored key | Keep provider calls destination/model allowlisted with user-configurable request/input/output budgets, fixed live probe content, token caps, and explicit opt-in; enforce strict content delivery and dependency review | Before live provider testing |
| Browser data loss | Historical OPFS mock-state recovery is not portable evidence for BrowserPod; active backup UI is removed | Add BrowserPod-specific versioned migration, recovery UX, and encrypted user-workspace backups before production-ready status | Phase 3 |
| Browser support overclaim | Browser-local runtime behavior differs across Chromium, Firefox, and Safari | Support desktop Chromium first and publish a provider-specific browser matrix based on test artifacts | Before 0.2 |
| Cross-version evidence reuse | A new dist-tag could otherwise appear verified using an older release's Gateway artifact | Match the embedded OpenClaw version before attachment and make the report builder reject all mismatches | Every generated report |
| Synthetic or cross-browser BrowserPod evidence | A structurally plausible readiness record could otherwise promote a different runtime build, browser, or npm artifact | Schema-check raw evidence, bind runtime version and exact browser string, compare installed package-lock SHA-512 to the inspected npm integrity, verify canonical digest, and recompute statuses | Every BrowserPod report |
| Upstream velocity | Stable releases are frequent | Start automated static inspection in Phase 0 and runtime inspection in Phase 1 | Continuous |
| Product sprawl | Embedded, remote, persistence, WIT, and channels compete for focus | Compatibility lab first; one-turn demo second; defer ecosystem features | Every milestone |
| Upstream confusion | Name and behavior may appear official | Keep the unofficial notice visible and seek concrete upstream review using evidence | Before launch announcement |

## Stop or pivot conditions

Pivot the project toward the Compatibility Lab and generated Gateway client if:

- a future stable cannot reach a Gateway handshake without changing agent logic;
- identity or provider secrets must be exposed to untrusted workspace code;
- no browser-local provider can support the intended commercial use;
- supported-browser boot performance makes the one-turn experience impractical.
